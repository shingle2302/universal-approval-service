## **一、实施目标与范围**

**目标**：在 Spring Boot + Flowable + MyBatis-Plus + H2（可平滑切换）基础上，交付一个可用于多业务线复用的通用审批服务，覆盖：

1. 模板管理（JSON 模板、版本、部署）
2. 流程执行（启动、任务、历史、监控）
3. 事件驱动（Spring Event + 可替换 MQ 抽象）
4. 复杂场景（回退、会签、动态变量、补偿）
5. 可靠性与可观测（本地消息、重试、审计）
6. 安全与权限（认证授权、接口权限控制）

***

## **二、总体里程碑（建议 12 周）**

- **M1（第 1-2 周）**：项目骨架 + 基础设施完成（可启动流程 Demo）
- **M2（第 3-4 周）**：模板中心 + BPMN 转换/部署闭环完成
- **M3（第 5-6 周）**：流程/任务核心 API 完成，支持主流程业务接入
- **M4（第 7-8 周）**：事件总线 + 本地消息可靠投递完成
- **M5（第 9-10 周）**：复杂场景（回退/会签/补偿/动态变量）完成
- **M6（第 11 周）**：安全、监控、运维能力补齐
- **M7（第 12 周）**：联调、压测、灰度上线与验收

***

## **三、分阶段实施计划（WBS）**

## **阶段 0：立项与方案冻结（第 1 周前半）**

- 输出《需求边界清单》：第一期必须支持的审批场景（如采购、订单、费用）
- 冻结关键技术版本（Spring Boot/Flowable/MyBatis-Plus/JDK）
- 定义 NFR 基线：性能、可用性、审计、RTO/RPO
- 确认接口规范（OpenAPI）、错误码、鉴权方式

**交付物**

- 需求范围说明书
- 技术选型与版本矩阵
- 架构评审纪要

***

## **阶段 1：工程基础建设（第 1-2 周）**

- 初始化模块分层：
  - **`api`**（controller + dto）
  - **`application`**（用例服务）
  - **`engine`**（Flowable 封装）
  - **`adapter`**（模板/回调路由）
  - **`event`**（发布订阅）
  - **`infrastructure`**（db、mapper、配置）
- 建库与迁移脚本（Flowable 原生表 + biz\_\* 扩展表）
- 全局异常、日志 traceId、审计日志骨架
- 完成健康检查、配置中心/环境区分（dev/test/prod）

**验收标准**

- 应用可启动，数据库自动初始化
- 一个最小流程可被部署并启动

***

## **阶段 2：模板中心与 BPMN 生成（第 3-4 周）**

- 实现 **`biz_approval_template`** 管理：
  - 创建/更新/查询/删除/版本化
- JSON 模板校验器：
  - 节点完整性、环路检查、条件表达式检查
- 实现 **`BpmnGeneratorService`**：
  - JSON -> BpmnModel -> 部署 Flowable
- 模板部署流水：
  - 保存模板版本 + deploymentId + processDefinitionKey

**验收标准**

- **`/api/approval/template/*`** 全部可用
- 模板可部署且在 Flowable 中可见
- 非法模板可被明确拦截并返回可读错误

***

## **阶段 3：流程执行与任务 API（第 5-6 周）**

- 流程管理 API：
  - 启动流程、查询实例、终止流程、变量更新
- 任务管理 API：
  - 待办列表、任务详情、完成任务、委派
- 审批日志 **`biz_approval_log`**：
  - 启动、审批、回退、结束全链路记录
- 历史查询 API：
  - process/task/log 历史聚合查询

**验收标准**

- 文档中 5.2/5.3/5.4 API 全通
- 可完成“发起->审批->结束”完整流程
- 审批轨迹可追溯（按 businessKey 可查）

***

## **阶段 4：事件驱动与可靠消息（第 7-8 周）**

- 定义统一事件模型 **`ApprovalEvent`**
- 实现 **`ApprovalMessagePublisher`** 抽象及 Spring 实现
- 集成流程监听器（流程启动/任务完成/流程结束/回退）
- **`biz_event_message`** 本地消息表：
  - PENDING/SUCCESS/FAILED 状态机
  - 重试任务（定时补偿）
- 业务路由机制：
  - 按 **`templateCode`** 分发到业务处理器

**验收标准**

- 核心事件可稳定发布与消费
- 消费失败后可重试并可观测
- 路由可支持至少 2 条业务线示例

***

## **阶段 5：复杂场景实现（第 9-10 周）**

- 回退能力：
  - 退上一步、退发起人、退指定节点
  - 回退时变量清理（**`autoResetVarsOnReject`**）
- 会签能力：
  - 并行/串行会签
  - 一人驳回全部退回策略
- 动态变量：
  - 表达式 Bean 注入
  - Service Task 拉取式数据补齐
- 自动补偿：
  - 触发补偿事件
  - **`biz_compensation_log`** 记录结果与错误

**验收标准**

- 三种回退路径均可用
- 会签驳回策略符合预期
- 补偿动作有日志闭环，可重放/审计

***

## **阶段 6：安全、监控与运维（第 11 周）**

- 鉴权与授权（建议接入企业 SSO/JWT）
- API 访问控制（角色、模板级权限）
- 指标与告警：
  - 启动数、完成数、平均处理时长、失败率
- 日志体系：
  - 业务日志、系统日志、审计日志
- 运维能力：
  - 失败消息重试管理接口
  - 补偿任务人工触发接口

**验收标准**

- 敏感接口全部受权限保护
- 关键指标可在监控面板查看
- 故障演练可完成“失败->重试->恢复”闭环

***

## **阶段 7：联调、压测、灰度上线（第 12 周）**

- 与业务系统端到端联调
- 场景压测（高并发发起审批、批量待办查询）
- 灰度发布（单业务线 -> 多业务线）
- 发布回滚预案演练

**验收标准**

- 关键 SLA 达标（如 P95、错误率）
- 灰度阶段无阻断问题
- 具备一键回滚与数据恢复方案

***

## **四、团队分工建议**

- **架构/Tech Lead（1）**：架构治理、关键设计评审、疑难问题攻坚
- **后端开发（3-5）**：
  - 1人模板中心/BPMN
  - 2人流程与任务 API
  - 1人事件与可靠消息
  - 1人复杂场景（回退/会签/补偿）
- **测试（1-2）**：功能、集成、回归、性能
- **运维/平台（1）**：部署流水线、监控告警、发布策略

***

## **五、关键风险与应对**

1. **Flowable 与数据库兼容风险**
   - 应对：先用 H2 File Mode 跑通；并行验证 MySQL/PostgreSQL 迁移脚本
2. **复杂回退/会签边界复杂**
   - 应对：先固化“白名单场景”，每个场景提供可回放测试用例
3. **事件一致性与重复消费**
   - 应对：本地消息表 + 幂等键（businessKey + eventType + version）
4. **表达式动态变量引发运行时错误**
   - 应对：模板发布前做表达式预校验 + 沙箱测试
5. **跨业务线接入成本高**
   - 应对：统一 SDK（请求/事件模型）+ 接入脚手架文档 + 示例项目

***

-  模板版本管理与回溯可用

-  回退/会签/补偿关键场景通过
-  审计日志可按业务主键追踪全流程
-  灰度及回滚预案验证通过
